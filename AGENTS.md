# Status Page — Agent Instructions

Uptime monitor and status page for [Reica](https://getreica.com), powered by [Upptime](https://github.com/upptime/upptime).  
Live: **https://status.getreica.com**

## What this repo is

Almost everything here is **auto-generated by GitHub Actions**. The source of truth is `.upptimerc.yml`. The only files a human (or agent) should ever edit are:

| File / Dir | Edit? | Notes |
|---|---|---|
| `.upptimerc.yml` | ✅ Yes | Add/remove monitored sites, change theme/CSS, set CNAME |
| `.github/workflows/` | ⚠️ Rarely | Only to change schedule or add custom steps |
| `assets/reica-theme.css` | ✅ Yes | Extra CSS extracted for readability |
| `history/*.yml` | ❌ Never | Auto-generated incident history |
| `api/**/*.json` | ❌ Never | Auto-generated metrics |
| `graphs/**` | ❌ Never | Auto-generated PNG graphs |
| `README.md` (status table) | ❌ Never | Auto-generated between `<!--start: status pages-->` markers |

## Monitored services

Defined in `.upptimerc.yml` under `sites:`. URLs are stored as **GitHub Secrets** (never hardcoded):

| Name | Secret | Notes |
|---|---|---|
| Reica Website | `$WEBSITE_URL` | Simple HTTP check |
| Product Service | `$PRODUCT_API_URL` | Requires `X-Health-Token` header |
| API Service | `$API_URL` | Requires `X-Health-Token` header |
| Relevance Score | `$RUNPOD_API_URL` | Accepts 200 or 401 as "up" |
| Database | `$SUPABASE_URL` | Accepts 200, 401, 404 as "up" |

## Adding a new service

1. Add the GitHub Secret for the URL in the repo settings.
2. Add an entry under `sites:` in `.upptimerc.yml` following the existing pattern.
3. Commit — GitHub Actions will handle the rest (history file, API JSON, graphs, README update).

## Theme / branding

Custom CSS lives inline in `.upptimerc.yml` under `status-website.css`. Brand tokens:
- Primary orange: `#c4622d`
- Background: `#f6f1ea`
- Font: Manrope (loaded via Google Fonts in `customHeadHtml`)

Do **not** inline external CSS or fonts outside of `customHeadHtml` / `css` fields.

## GitHub Actions workflows

| Workflow | Trigger | Purpose |
|---|---|---|
| `uptime.yml` | Scheduled | Checks site status, creates/closes incidents |
| `response-time.yml` | Scheduled | Records response time data |
| `graphs.yml` | Scheduled | Generates PNG graphs |
| `summary.yml` | Scheduled | Updates `history/summary.json` |
| `site.yml` | Scheduled + push | Deploys static status page to GitHub Pages |
| `updates.yml` | Scheduled | Upptime self-update |
| `setup.yml` | Manual | Initial repo setup |
| `update-template.yml` | Scheduled | Syncs with upstream Upptime template |

Never trigger `setup.yml` on an already-configured repo — it overwrites config.

## Common pitfalls

- **Do not edit auto-generated files** — every workflow run will overwrite them.
- **Secrets not env vars** — URLs in `.upptimerc.yml` use `$SECRET_NAME` syntax, resolved at runtime by Upptime. Do not wrap them in quotes or double-dollar signs.
- **`expectedStatusCodes`** — services that return 401/404 on the health endpoint (e.g., Supabase, RunPod) must list those codes explicitly, otherwise Upptime marks them as down.
- **CNAME** — `status-website.cname` in `.upptimerc.yml` must match the DNS CNAME record. Changing it without updating DNS will break the page.
